Esta documentacion esta escrita como guia de estudio personal para preparar la presentacion del proyecto.

La idea no es describir cada funcion, sino darte una comprension global y profunda del sistema:

  • como se divide

  • como se comunican las piezas

  • donde esta la logica realmente importante

  • como funciona el online

  • que decisiones de arquitectura hay detras

  • que limites tiene la implementacion actual

1. Vision General

1.1. Que es Yovi

Yovi es una aplicacion web para jugar al Game Y, un juego de conexion sobre tablero triangular.

La aplicacion ofrece cuatro capacidades principales:

  • registro e inicio de sesion

  • partidas contra bots

  • partidas humanas en el mismo dispositivo

  • partidas humanas online con emparejamiento

  • almacenamiento de historial y estadisticas

Si tuviera que resumir el proyecto en una sola frase para una presentacion, seria esta:

Yovi es una SPA que entra por un gateway y reparte responsabilidades entre servicios especializados: autenticacion, motor de juego y estadisticas.

1.2. Mapa mental del sistema

La mejor manera de entenderlo es imaginar tres capas:

  1. Capa de acceso

    El navegador carga la SPA y todas las peticiones entran por el gateway.

  2. Capa de servicios

    Aqui viven auth_service, gamey y stats.

  3. Capa de persistencia y operacion

    Dos Mongo separadas guardan autenticacion y estadisticas. Prometheus y Grafana observan el sistema.

1.3. Componentes principales

1.3.1. Webapp

Es la interfaz de usuario.

Responsabilidades:

  • mostrar login, dashboard, partida, historial y ayuda

  • guardar sesion local del usuario

  • guardar informacion suficiente para reanudar partidas o matchmaking

  • hacer peticiones HTTP a backend

  • traducir el estado de juego a una representacion visual del tablero

No contiene las reglas reales del juego. La autoridad del estado esta en gamey.

1.3.2. Gateway

Es la puerta de entrada publica.

Responsabilidades:

  • recibir el trafico externo

  • reenviar cada prefijo al servicio correcto

  • servir la webapp como punto de entrada unico

  • exponer una API externa mas amigable bajo /external/*

  • normalizar errores de servicios caidos como 502 Bad Gateway

  • manejar HTTP o HTTPS segun configuracion

No deberia tener casi logica de negocio. Su papel es de fachada y orquestacion ligera.

1.3.3. Auth Service

Es el servicio de identidad.

Responsabilidades:

  • registrar usuarios

  • verificar credenciales

  • emitir JWT

  • validar JWT

  • guardar usuarios y hashes de contrasena en Mongo

1.3.4. Gamey

Es el nucleo del proyecto.

Responsabilidades:

  • aplicar reglas del Game Y

  • crear partidas

  • mantener partidas activas en memoria

  • ejecutar bots

  • gestionar matchmaking

  • supervisar tiempos de turno e inactividad

  • reportar partidas finalizadas a stats

Si te preguntan cual es el microservicio mas importante, es gamey.

1.3.5. Stats

Es el servicio de historial y estadisticas.

Responsabilidades:

  • devolver el resumen del jugador

  • devolver historial de partidas terminadas

  • recibir internamente los resultados finales desde gamey

  • actualizar contadores agregados

  • guardar snapshot del tablero final y datos de la partida

1.4. Tecnologias y por que encajan

1.4.1. Frontend

React + TypeScript + Vite + MUI

Encaja porque:

  • permite una SPA rapida y reactiva

  • facilita dividir la interfaz en vistas y hooks

  • hace sencillo persistir estado en navegador

1.4.2. Gateway, auth y stats

Node.js + Express

Encaja porque:

  • son servicios HTTP relativamente ligeros

  • tienen poca carga computacional

  • la mayor parte del trabajo es I/O, no calculo pesado

Importante para explicarlo bien:

  • estos servicios no montan manualmente hilos de trabajo

  • siguen el modelo habitual de Node.js: bucle de eventos y operaciones asincronas

1.4.3. Gamey

Rust + Axum + Tokio

Encaja porque:

  • el motor del juego y los bots concentran la logica mas sensible

  • Rust permite controlar muy bien estado, concurrencia y rendimiento

  • Tokio facilita lanzar tareas asincronas de fondo para matchmaking y supervision

Aqui si aparece una idea mas cercana a "workers" o "bucles periodicos":

  • una tarea asincrona revisa la cola de matchmaking

  • otra tarea asincrona revisa los timeouts de partidas online

1.4.4. Persistencia

MongoDB

Se usan dos bases separadas:

  • mongo-auth para identidades

  • mongo-stats para historial y agregados

La separacion refuerza la independencia entre dominios.

1.5. Recorrido rapido de una sesion completa

1.5.1. 1. El usuario entra

  • el navegador accede al gateway

  • el gateway sirve la SPA

  • la SPA comprueba si ya hay token valido

1.5.2. 2. El usuario juega

  • la SPA pide crear partida o entrar en matchmaking

  • el gateway reenvia a gamey

  • gamey crea o recupera el estado de la partida

1.5.3. 3. La partida avanza

  • en local o contra bot, la SPA hace peticiones puntuales

  • en online, la SPA va sincronizando el estado por polling

  • gamey actualiza el estado en memoria

1.5.4. 4. La partida termina

  • gamey detecta ganador, rendicion o abandono

  • prepara un resumen de fin de partida

  • lo envia a stats

1.5.5. 5. El historial se consulta despues

  • la SPA pide estadisticas e historial

  • stats lee Mongo y devuelve agregados y partidas terminadas

1.6. Idea fuerza de la arquitectura

La arquitectura esta pensada para separar muy bien tres tipos de responsabilidad:

  • quien eres: auth_service

  • que esta pasando ahora mismo en una partida: gamey

  • que ha pasado historicamente: stats

El gateway une todo eso bajo una sola entrada publica.

1.7. Donde esta la complejidad real

La complejidad del proyecto no esta repartida por igual.

Se concentra sobre todo en estas zonas:

  • gamey, porque mezcla reglas, bots, sesiones activas, matchmaking y timeouts

  • useGamey en frontend, porque coordina restauracion de estado, polling y persistencia local

  • el cruce gamey → stats, porque separa estado vivo de historial persistente

En cambio, auth_service y stats son mucho mas directos: reciben datos, validan, leen o escriben en Mongo y responden.

2. Microservicios

2.1. Vision conjunta

Los microservicios del proyecto no son todos del mismo tipo.

Hay una division bastante clara:

  • gateway: fachada publica

  • auth_service: identidad

  • gamey: dominio principal y tiempo real simplificado

  • stats: lectura historica y persistencia de resultados

Esta separacion tiene sentido porque corta el sistema por responsabilidad y no por capas artificiales.

2.2. Gateway

2.2.1. Rol arquitectonico

El gateway es el unico punto de entrada publico del sistema.

Explicado de forma sencilla:

  • el navegador no conoce la topologia interna

  • sabe hablar con un solo servidor

  • ese servidor decide a quien reenviar cada peticion

Esto simplifica mucho la explicacion externa del sistema.

2.2.2. Que expone

Rutas principales:

  • / hacia la webapp

  • /auth/* hacia auth_service

  • /api/* hacia gamey

  • /stats/* hacia stats

  • /external/* como API publica agregada

  • /health y /metrics para operacion

2.2.3. Como trabaja internamente

Internamente es un servicio fino:

  • Express recibe la peticion

  • http-proxy-middleware la reenvia al destino correcto

  • si un servicio aguas abajo no responde, el gateway transforma el fallo en un 502

Eso es importante para presentarlo:

  • centraliza el acceso

  • no centraliza la logica de negocio

2.2.4. API externa agregada

Ademas del proxy normal, el gateway tiene un bloque especial bajo /external.

Ese bloque no solo reenvia, sino que:

  • publica documentacion Swagger

  • expone un catalogo publico de bots

  • ofrece endpoints unificados para usuarios y partidas

  • verifica JWT en ciertos endpoints agregados

Esto le da un segundo papel:

  • proxy transparente para la SPA

  • fachada API para consumidores externos

2.2.5. HTTPS y redireccion

El gateway puede trabajar:

  • en HTTP plano

  • en HTTPS si se le montan certificado y clave

  • con redireccion opcional de HTTP a HTTPS

Eso lo convierte en el borde publico del despliegue.

2.2.6. Observabilidad

Expone metricas Prometheus sobre:

  • numero de peticiones

  • tiempos agregados de respuesta

  • uptime

  • memoria del proceso

2.3. Auth Service

2.3.1. Rol arquitectonico

auth_service resuelve todo lo relacionado con identidad, pero solo eso.

No sabe nada del tablero, del matchmaking ni del historial de partidas.

2.3.2. Endpoints y comportamiento

Expone tres rutas clave:

  • POST /register

  • POST /login

  • GET /verify

Flujo interno:

  1. valida entrada

  2. consulta Mongo

  3. en registro, crea hash con bcrypt

  4. en login, compara hash

  5. si todo va bien, firma un JWT

2.3.3. Modelo de datos

En Mongo guarda usuarios con:

  • username

  • passwordHash

  • createdAt

Detalle importante:

  • el username se normaliza a minusculas

  • la contrasena nunca se guarda en claro

2.3.4. Seguridad

Fortalezas reales:

  • usa bcrypt para hashing

  • usa JWT para sesion

  • tiene endpoint de verificacion de token

  • no comparte la base de datos de auth con otros dominios

Matiz importante para explicar:

  • el servicio si es el responsable autentico de credenciales

  • pero la SPA no basa toda la aplicacion en reenviar el JWT a todos los servicios en todo momento

Ese matiz aparece mejor en la seccion de datos e identidades.

2.3.5. Modelo de ejecucion

Como servicio Node.js:

  • no lanza hilos manuales

  • funciona sobre el bucle de eventos

  • maneja peticiones HTTP y acceso a Mongo de forma asincrona

2.4. Gamey

2.4.1. Rol arquitectonico

gamey es el corazon de Yovi.

Es a la vez:

  • motor del juego

  • API de partidas

  • ejecutor de bots

  • gestor de matchmaking

  • supervisor de partidas online

  • emisor de eventos finales hacia stats

Si tuvieras que justificar por que este servicio esta separado, la respuesta es:

porque concentra la logica de dominio que cambia el estado real del juego y necesita control fino de concurrencia y consistencia

2.4.2. Subbloques internos

Dentro de gamey se distinguen muy bien varios bloques:

  • core: reglas del Game Y

  • bot: bots y registro de estrategias

  • bot_server/mod.rs: router HTTP Axum

  • bot_server/state.rs: estado compartido de la aplicacion

  • bot_server/games.rs: endpoints y supervision de partidas

  • bot_server/matchmaking.rs: cola, tickets y worker de emparejamiento

  • bot_server/choose.rs: endpoint para pedir jugadas a bots

  • bot_server/metrics.rs: metricas Prometheus

2.4.3. El motor del juego

El motor no trabaja con componentes visuales, sino con un modelo abstracto.

Ideas clave para explicar:

  • el tablero triangular se representa con coordenadas barycentricas x, y, z

  • una celda valida cumple x + y + z = N - 1

  • cada jugador va colocando fichas por turnos

  • gana quien conecta los tres lados del triangulo con una sola componente conectada

La deteccion de victoria esta optimizada con una idea muy buena para presentar:

  • cada grupo conectado de fichas de un jugador se trata como un conjunto

  • ese conjunto recuerda si toca el lado A, B o C

  • cuando dos grupos vecinos se unen, se combinan esas marcas

  • si un conjunto acaba tocando los tres lados, hay victoria

En otras palabras:

  • no hace falta recalcular todo el tablero desde cero en cada movimiento

  • el motor mantiene informacion estructural sobre componentes conectadas

2.4.4. Bots

Hay cuatro niveles de bot registrados por nombre:

  • random_bot

  • biased_random_bot

  • greedy_bot

  • minimax_bot

La dificultad que ve el usuario en frontend se traduce a esos identificadores.

Conceptualmente:

  • random elige una jugada valida al azar

  • biased_random mete sesgo posicional

  • greedy puntua estados y escoge una opcion localmente buena

  • minimax mira mas alla y es la estrategia mas competitiva

El registro de bots permite:

  • pedir un bot por nombre

  • cambiar bot sin tocar el resto del flujo

  • exponer un catalogo claro hacia fuera

2.4.5. Estado compartido

AppState es la pieza central de concurrencia dentro de gamey.

Guarda:

  • registro de bots

  • mapa de partidas activas

  • indice de partida activa por usuario

  • estado del matchmaking

  • metricas

  • contadores atomicos para generar ids y tokens

Detalles arquitectonicos importantes:

  • el estado compartido usa Arc, por lo que varias partes del servidor pueden referenciarlo sin copiarlo entero

  • los mapas mutables importantes van dentro de RwLock

  • los identificadores se generan con contadores atomicos

Eso te permite explicar muy bien como se combina concurrencia con seguridad:

  • varias tareas comparten el mismo estado

  • se sincroniza acceso a escritura

  • no hay una base externa para el estado vivo

2.4.6. Modos de juego

Hay tres formas reales de jugar que conviene diferenciar:

  1. human_vs_bot

    La partida vive en gamey. Tras la jugada humana, gamey puede resolver en la misma peticion la respuesta del bot.

  2. human_vs_human local

    No usa matchmaking ni tokens. Sigue siendo una partida en gamey, pero pensada para alternar jugadores sin coordinacion remota especial.

  3. human_vs_human online con matchmaking

    Aqui aparece la parte mas rica del servicio: tickets, tokens por jugador, supervisores y polling.

2.4.7. Creacion de partidas

Cuando se crea una partida:

  • se valida el tamano de tablero

  • se resuelve el bot si procede

  • se leen identidades desde cabeceras

  • se comprueba si esos usuarios ya tienen otra partida activa

  • se crea una GameSession

  • se guarda en memoria bajo un game_id

Importante:

  • no se escribe nada en base de datos al crear

  • el estado vivo reside en memoria del proceso

2.4.8. Endpoints principales

Rutas relevantes:

  • POST /v1/games

  • GET /v1/games/{id}

  • POST /v1/games/{id}/moves

  • POST /v1/games/{id}/resign

  • POST /v1/games/{id}/pass

  • POST /v1/matchmaking/enqueue

  • GET /v1/matchmaking/tickets/{id}

  • POST /v1/matchmaking/tickets/{id}/cancel

  • POST /v1/ybot/choose/{bot_id}

2.4.9. Validacion y autoridad

gamey es quien decide si una accion es valida.

Ejemplos de decisiones que toma:

  • si la coordenada esta dentro del tablero

  • si la celda esta libre

  • de quien es el turno

  • si el token del jugador online corresponde al turno actual

  • si la partida ya esta terminada

La SPA puede mostrar ayudas, temporizadores y botones, pero la autoridad final es gamey.

2.4.10. Matchmaking

El matchmaking mantiene dos estructuras:

  • una cola FIFO de espera

  • un mapa de tickets con estado waiting, matched o cancelled

Cada ticket recuerda:

  • tamano del tablero

  • identidad del usuario o invitado

  • momento de entrada

Un worker asincrono corre periodicamente y:

  • mira si hay dos tickets compatibles

  • evita emparejar la misma identidad consigo misma

  • crea la partida online

  • genera un token por jugador

  • transforma ambos tickets a matched

2.4.11. Supervision online

Ademas del worker de matchmaking, gamey lanza otro bucle de fondo que revisa partidas online cada segundo.

Ese monitor aplica dos reglas:

  • si un jugador lleva mas de 60 segundos sin dar senales, pierde por abandono

  • si un turno supera 60 segundos, se fuerza un pass

Esto es una de las ideas mas potentes del proyecto porque muestra backend activo, no solo un CRUD.

2.4.12. Integracion con stats

Cuando una partida termina:

  • gamey detecta ganador y motivo

  • construye un resumen con modo, bot, ganador, tablero final y lista de jugadores

  • lo envia por HTTP a stats

Muy importante:

  • el envio de estadisticas no bloquea la respuesta de juego de forma dura

  • si stats falla, el jugador igualmente recibe el estado final de la partida

Eso es una decision de resiliencia.

2.5. Stats

2.5.1. Rol arquitectonico

stats separa el mundo del estado vivo del mundo de la memoria historica.

No participa en la jugada actual.

Solo trabaja con partidas terminadas y consultas posteriores.

2.5.2. Endpoints

Publicos de lectura:

  • GET /v1/me

  • GET /v1/me/history

Interno de escritura:

  • POST /internal/v1/matches/finished

2.5.3. Modelo de persistencia

Usa dos colecciones principales:

  • player_stats

  • player_matches

La idea es separar:

  • agregados rapidos por jugador

  • historial detallado por partida

2.5.4. Como procesa un fin de partida

Cuando gamey llama al endpoint interno:

  1. stats valida el payload

  2. recorre la lista de jugadores

  3. inserta la partida de cada jugador si no existia

  4. incrementa contadores agregados

Este punto es importante:

  • la escritura esta pensada para ser idempotente

  • usa upsert

  • hay indices unicos para evitar duplicados

Eso significa que si por alguna razon se intenta reportar dos veces la misma partida, no se deberian inflar las estadisticas.

2.5.5. Que guarda de cada partida

Guarda cosas muy utiles para el historial:

  • gameId

  • userId

  • result

  • mode

  • botId

  • reason

  • winnerId

  • finalBoard

  • endedAt

Gracias a finalBoard, el frontend puede enseñar una previsualizacion del estado final sin volver a consultar a gamey.

2.5.6. Seguridad interna

El endpoint de escritura interna pide x-service-token.

Eso evita que un cliente cualquiera escriba resultados arbitrarios en historial solo con llamar al endpoint.

2.5.7. Modelo de ejecucion

Igual que auth_service, es un servicio Node.js orientado a I/O:

  • Express para HTTP

  • driver oficial de Mongo

  • metricas de proceso y peticiones

2.6. Lo que esta bien separado y lo que no

2.6.1. Separaciones bien resueltas

  • credenciales y cuentas viven fuera del motor de juego

  • historial y estadisticas viven fuera del motor de juego

  • el gateway evita exponer topologia interna

  • gamey concentra reglas y coordinacion online

2.6.2. Acoplamientos que existen

Hay tambien algunos acoplamientos reales, utiles de reconocer:

  • gamey depende de stats para persistir resultados finales, aunque degrade con elegancia si falla

  • la SPA conoce contratos concretos de gamey y stats

  • el online depende de que el estado de gamey siga vivo en memoria

Saber reconocer estas dependencias te hace sonar mucho mas solido en una presentacion.

3. Online y Matchmaking

3.1. La idea central

El online de Yovi no esta construido con WebSockets ni con un sistema de eventos en tiempo real empujados desde servidor.

Esta construido con una combinacion de:

  • polling HTTP desde el navegador

  • estado en memoria dentro de gamey

  • workers asincronos en segundo plano

  • tokens de jugador para identificar quien puede actuar

Esta idea es fundamental para explicarlo bien:

no es tiempo real push, es sincronizacion periodica controlada

3.2. Piezas que participan

Para una partida online intervienen estas piezas:

  • DashboardView y useGamey en frontend

  • gameSessionStore en navegador

  • gateway

  • endpoints de matchmaking y juegos de gamey

  • worker de emparejamiento

  • monitor de timeouts

3.3. Paso 1. El usuario pulsa "Buscar rival"

Cuando el usuario inicia matchmaking:

  • la SPA llama a POST /api/v1/matchmaking/enqueue

  • la peticion llega a gamey

  • gamey crea un ticket_id

El ticket se mete en dos sitios:

  • en una cola FIFO

  • en un mapa de tickets con estado waiting

Ademas se guarda informacion de contexto:

  • tamano del tablero

  • identidad del usuario o sesion invitada

  • instante de entrada

Detalle importante:

  • si la identidad ya tiene otra partida activa, el servicio rechaza la operacion

  • si la identidad ya tiene un ticket esperando, tambien se rechaza

3.4. Paso 2. La SPA empieza a hacer polling del ticket

Nada mas recibir el ticket:

  • el frontend guarda ticket_id y estado waiting

  • persiste esa informacion en localStorage

  • programa un temporizador para volver a consultar el ticket

La consulta es:

  • GET /api/v1/matchmaking/tickets/{ticket_id}

La frecuencia base de polling es aproximadamente de 1 segundo.

Esto es importante:

  • el navegador no queda bloqueado

  • no necesita una conexion viva permanente

  • si recargas la pagina, la app puede recuperar la espera desde localStorage

3.5. Paso 3. Worker de matchmaking en gamey

Mientras el cliente esta preguntando, gamey tiene un worker asincrono propio corriendo en bucle.

Comportamiento del worker:

  • despierta cada 300 ms

  • intenta sacar dos tickets compatibles de la cola

  • exige mismo tamano de tablero

  • evita emparejar la misma identidad consigo misma

Cuando encuentra pareja:

  • crea un game_id

  • genera un token distinto para cada jugador

  • crea una GameSession online en memoria

  • marca los tickets como matched

Este es un muy buen punto para defender arquitectura:

  • no se usa una cola externa

  • no se usa una base de datos compartida para matchmaking

  • el emparejamiento se resuelve en memoria con un worker periodico sencillo

3.6. Paso 4. El ticket pasa a matched

En cuanto el worker empareja:

  • cada ticket deja de estar waiting

  • ahora incluye game_id

  • ahora incluye player_id

  • ahora incluye player_token

La siguiente vez que el navegador hace polling:

  • ve que ya hay partida

  • deja de consultar el ticket

  • abre automaticamente la partida

3.7. Paso 5. Se abre la partida online

En ese momento el frontend:

  • pide GET /api/v1/games/{game_id}

  • incluye el x-player-token

  • guarda localmente que esa sesion online esta activa

La sesion persistida en navegador recuerda:

  • gameId

  • myPlayerId

  • playerToken

Esto es lo que hace posible reanudar una partida online despues de recargar.

3.8. Paso 6. Sincronizacion continua de la partida

Una vez abierta la partida online, el frontend entra en un segundo modo de polling.

Hace consultas periodicas a la partida activa:

  • aproximadamente cada 1.2 segundos

Cada sincronizacion sirve para:

  • refrescar el tablero

  • ver de quien es el turno

  • actualizar cuenta atras

  • detectar si el rival ha jugado

  • saber si la partida ha terminado

Es decir:

  • no hay push del servidor

  • el cliente pregunta y se sincroniza

3.9. Paso 7. Que protege al online

La proteccion del online no depende solo de saber el game_id.

Cada partida de matchmaking genera player_token por jugador.

Ese token se usa para:

  • leer la partida como jugador online identificado

  • jugar movimientos

  • pasar turno

  • rendirse

gamey valida que:

  • el token exista

  • corresponda a uno de los jugadores de esa partida

  • en ciertas acciones, corresponda exactamente al jugador cuyo turno toca

Ese detalle es importantisimo:

  • el game_id localiza la partida

  • el player_token acredita la identidad operativa dentro de esa partida

3.10. Paso 8. Que pasa cuando alguien juega

Cuando el jugador pulsa una celda:

  • la SPA envia POST /api/v1/games/{id}/moves

  • manda coordenadas

  • si es online, el backend valida token y turno

Entonces gamey:

  1. valida coordenadas

  2. valida turno

  3. actualiza el motor del juego

  4. reinicia temporizador de turno si procede

  5. si termina la partida, prepara reporte a stats

Si la partida es contra bot, ademas:

  • tras la jugada humana calcula y aplica inmediatamente la jugada del bot en la misma operacion logica

3.11. Paso 9. Presencia e inactividad

Cada vez que un jugador online consulta o actua sobre la partida:

  • gamey actualiza su last_seen

Ese dato no se guarda en base de datos.

Se guarda en memoria como parte de la GameSession.

Esto permite detectar:

  • jugador activo

  • jugador ausente

  • rival que lleva demasiado tiempo sin volver

3.12. Paso 10. Monitor de timeouts

Ademas del worker de matchmaking, hay otro bucle asincrono de fondo.

Ese monitor:

  • se despierta cada 1 segundo

  • revisa todas las partidas online activas en memoria

Aplica dos reglas:

3.12.1. Regla A. Timeout de inactividad del jugador

Si un jugador no da senales durante mas de 60 segundos:

  • el sistema lo considera desconectado

  • la partida termina por disconnect_timeout

  • tecnicamente se resuelve como una rendicion forzada

3.12.2. Regla B. Timeout de turno

Si el turno actual supera 60 segundos:

  • el monitor ejecuta un pass automatico

  • el turno cambia

Esto te permite explicar que el online tiene:

  • supervision en backend

  • no solo temporizadores cosmeticos en frontend

3.13. Lo que hace tambien el frontend con los temporizadores

La interfaz no se limita a mostrar lo que manda el servidor.

Tambien:

  • sincroniza una cuenta atras visual

  • la actualiza en local cada 250 ms

  • en ciertos casos dispara automaticamente passCurrentTurn() cuando la cuenta llega a cero

Interpretacion correcta para explicarlo:

  • el backend sigue siendo la autoridad real

  • el frontend mejora la experiencia visual y reduce sensacion de espera

  • hay una cooperacion entre la cuenta atras local y la supervision real de gamey

3.14. Recuperacion tras recarga o salida accidental

Este es uno de los detalles mejor resueltos de la SPA.

useGamey intenta restaurar:

  • un matchmaking en curso

  • una partida local activa

  • una partida online activa

Lo hace leyendo localStorage.

Escenarios:

3.14.1. Si habia un ticket esperando

  • recupera el ticket_id

  • vuelve a consultar su estado

  • si sigue esperando, reanuda polling

  • si ya esta emparejado, abre la partida

3.14.2. Si habia una partida online activa

  • recupera gameId, myPlayerId y playerToken

  • vuelve a pedir el estado a gamey

  • si sigue activa, la reabre

3.14.3. Si el servidor ya no tiene ese estado

Puede fallar la restauracion si gamey ya no conserva el ticket o la partida.

Esto pasa porque:

  • el estado vivo esta en memoria

  • si el proceso se reinicia, ese estado desaparece

3.15. Como se cierra el flujo online

Cuando la partida termina:

  • gamey determina ganador y motivo

  • limpia el indice de partidas activas por usuario

  • conserva la sesion terminada en su mapa en memoria

  • envia resumen a stats

  • la SPA puede refrescar estadisticas si el usuario esta autenticado

3.16. Lo que debes decir si te preguntan "como funciona el online"

Respuesta corta y fuerte:

El online de Yovi no usa WebSockets. Usa matchmaking y sesiones en memoria dentro de gamey, polling periodico desde la SPA, tokens por jugador para validar acciones y dos workers asincronos: uno empareja jugadores y otro supervisa inactividad y tiempo de turno.

Respuesta mas larga:

Cuando el usuario entra en cola, gamey crea un ticket y un worker lo revisa cada pocos cientos de milisegundos hasta emparejarlo con otro compatible. Una vez creada la partida, cada jugador recibe un token propio. El navegador consulta periodicamente el estado de esa partida y gamey va actualizando la presencia de cada jugador. En paralelo, otro worker recorre las partidas online para forzar un pass por exceso de tiempo o cerrar la partida por abandono. Al finalizar, gamey envia el resumen a stats para que el historial quede persistido.

3.17. Fortalezas de este enfoque

  • es sencillo de entender y depurar

  • evita complejidad de sockets o broker de eventos

  • separa bien estado vivo y estado historico

  • permite reanudar sesion desde navegador

  • mantiene gamey como autoridad unica del estado

3.18. Costes de este enfoque

  • hay mas trafico HTTP por polling

  • el estado vivo depende de la memoria del proceso

  • si quisieras escalar horizontalmente gamey, ya no bastaria con otra replica sin mas

  • la experiencia no es tan inmediata como un sistema push real

Saber contar tanto fortalezas como costes te da mucha credibilidad.

4.1. Idea principal

Una de las claves de este proyecto es que no todo el estado vive en el mismo sitio.

Hay al menos cuatro niveles distintos:

  • estado operativo casi nulo en gateway

  • estado vivo de partidas en gamey

  • estado persistente en Mongo

Si entiendes esto, entiendes medio proyecto.

4.3. Estado en el navegador

4.3.1. Autenticacion

El frontend guarda en localStorage:

  • authToken

  • authUsername

  • guestSessionId

Con eso puede distinguir:

  • usuario autenticado

  • invitado

  • usuario sin sesion

Cuando la app arranca:

  • si hay token, llama a /auth/verify

  • si el token ya no vale, lo limpia

4.3.2. Sesion de juego reanudable

La SPA tambien guarda una estructura propia de sesion de juego.

Puede ser de tres tipos:

  • online_waiting

  • online_active

  • local_active

Eso significa que el navegador no solo recuerda "que usuario soy", sino tambien:

  • si estoy esperando rival

  • si tengo una partida activa

  • con que token debo reanudar una partida online

4.3.3. Que gana la app con esto

Gana dos cosas:

  • mejor experiencia de usuario

  • menos fragilidad frente a recargas accidentales

Pero no es una persistencia total del juego.

El navegador solo guarda metadatos para reconectar. El estado real sigue estando en gamey.

4.4. Estado en el gateway

El gateway es casi sin estado.

No guarda:

  • partidas

  • usuarios

  • historial

Lo que si conserva es estado tecnico ligero:

  • configuracion de rutas

  • metricas

  • configuracion TLS

Eso es bueno porque hace facil entenderlo y sustituirlo.

4.5. Estado persistente de autenticacion

auth_service guarda usuarios en mongo-auth.

Cada usuario tiene:

  • nombre

  • hash de contrasena

  • fecha de creacion

La base de auth no se usa para guardar partidas ni historial.

Esto te permite decir que la identidad esta aislada del dominio de juego.

4.6. Estado vivo en gamey

4.6.1. Partidas activas

Cada partida vive como GameSession en memoria.

La sesion contiene, entre otras cosas:

  • el motor GameY

  • el bot si existe

  • instante de creacion

  • instante de inicio de turno

  • tokens por jugador en partidas online

  • ultima presencia de cada jugador online

  • ids de usuarios asociados

  • indicador de si ya se reporto a stats

  • motivo de finalizacion

Es un estado muy rico y muy orientado a ejecucion.

4.6.2. Indice de partida activa por usuario

Ademas del mapa principal de partidas, gamey mantiene otro mapa:

  • usuario normalizado → game_id activo

Eso se usa para impedir:

  • crear varias partidas activas simultaneas con la misma identidad

  • entrar varias veces en matchmaking con la misma identidad

4.6.3. Cola y tickets de matchmaking

El matchmaking no se deduce de las partidas activas.

Tiene su propio estado:

  • cola FIFO

  • mapa de tickets

Esto es importante porque separa dos fases distintas:

  • esperando rival

  • jugando partida

4.6.4. Partidas terminadas en memoria

Hay un detalle importante que conviene conocer:

  • las partidas terminadas no se eliminan automaticamente del mapa en memoria de gamey

Que implica esto:

  • la respuesta final se puede seguir consultando desde gamey

  • las metricas distinguen partidas en curso y partidas finalizadas en memoria

  • a largo plazo, si el proceso viviera muchisimo tiempo, habria crecimiento de memoria

Esto es una limitacion real de la implementacion actual.

4.7. Estado persistente en stats

4.7.1. Coleccion player_stats

Guarda el agregado por usuario:

  • totalGames

  • victories

  • defeats

  • updatedAt

Sirve para pintar el resumen rapido del dashboard y del historial.

4.7.2. Coleccion player_matches

Guarda una fila por usuario + partida.

Se conserva:

  • identificador de partida

  • modo

  • resultado

  • ganador

  • bot si aplica

  • motivo de finalizacion

  • tablero final

  • fecha de fin

Gracias a eso el historial puede:

  • listar partidas

  • filtrar por tipo

  • ordenar por fecha

  • abrir una vista del tablero final

4.7.3. Indices y consistencia

stats crea indices para:

  • un usuario tenga un solo documento agregado

  • una combinacion gameId + userId no se inserte dos veces

  • recuperar historial por usuario ordenado por fecha

Arquitectonicamente esto esta bien resuelto porque:

  • permite lecturas rapidas

  • evita contar dos veces una partida

4.8. Identidades: punto sutil e importante

Hay un matiz que merece que lo conozcas muy bien.

No toda la aplicacion usa exactamente la misma representacion de identidad en todos los caminos.

4.8.1. Camino de la SPA

La SPA trabaja sobre todo con:

  • username cuando el usuario esta autenticado

  • guestSessionId cuando entra como invitado

Esos valores viajan en cabeceras como x-user-id hacia gamey y stats.

4.8.2. Camino de la API externa agregada

La API bajo /external/* si verifica JWT y puede propagar el id verificado por auth_service.

4.8.3. Como explicarlo sin liarte

La forma mas limpia de decirlo es:

la aplicacion mezcla una identidad de sesion orientada a frontend con una identidad autenticada validada por auth, segun el camino de acceso que se utilice

No hace falta presentarlo como desastre, pero si como un detalle arquitectonico a tener en cuenta.

4.9. Por que stats solo se actualiza al final

No se registra cada movimiento en base de datos.

La razon es sencilla:

  • gamey esta optimizado para juego vivo en memoria

  • stats esta optimizado para memoria historica

Ventajas:

  • menos escrituras

  • menos latencia en jugadas

  • mejor separacion de responsabilidades

Coste:

  • no tienes replay persistente movimiento a movimiento

  • si gamey se cae antes de acabar, la partida viva no pasa al historial

4.10. Que pasa si se reinicia un servicio

4.10.1. Si se reinicia auth_service

  • se pierde el proceso, no la base de usuarios

  • mientras el JWT siga siendo valido y use la misma clave, la sesion puede verificarse otra vez

4.10.2. Si se reinicia stats

  • se pierde el proceso, no la base historica

  • podrian fallar temporalmente reportes de fin de partida si gamey intenta escribir justo en ese momento

4.10.3. Si se reinicia gamey

Este es el caso mas delicado:

  • se pierden partidas activas en memoria

  • se pierden tickets de matchmaking

  • se pierden tokens de jugador

  • la SPA puede tener datos para intentar restaurar, pero el servidor ya no conoceria esa partida

Este es uno de los mejores ejemplos para explicar por que el estado vivo en memoria simplifica mucho, pero penaliza resiliencia ante reinicios.

5.1. Despliegue base con Docker Compose

La forma natural de ejecutar el sistema es con docker-compose.

Contenedores principales:

  • webapp

  • gateway

  • auth

  • gamey

  • stats

  • mongo-auth

  • mongo-stats

  • prometheus

  • grafana

Todos comparten una red Docker interna.

5.2. Que expone cada contenedor

5.2.1. Webapp

  • se construye con Node

  • se sirve en produccion con nginx

  • publica solo contenido estatico

En el contenedor:

  • nginx sirve index.html

  • usa try_files para devolver la SPA tambien en rutas internas del cliente

Esto es importante si te preguntan como se sirve la parte frontend en Docker:

se compila a estaticos y luego nginx los sirve

5.2.2. Gateway

Es el verdadero borde publico.

Puertos:

  • HTTP en 8080 por defecto

  • HTTPS configurable, normalmente 443 en host y 8443 interno

Solo el gateway deberia exponerse como entrada web publica del sistema completo.

5.2.3. Auth

Expone 3500.

Depende de mongo-auth.

5.2.4. Stats

Expone 3001.

Depende de mongo-stats.

5.2.5. Gamey

Expone 4000.

No usa base de datos propia para partidas vivas.

  • Prometheus en 9090

  • Grafana en 9091

5.3. Flujo de red

En despliegue normal:

  1. el navegador entra por gateway

  2. el gateway reenvia al servicio correcto

  3. los servicios se hablan entre ellos por HTTP dentro de Docker

  4. auth y stats hablan con sus Mongo por driver de base de datos

5.11. Decisiones de operacion que se entienden bien

5.11.1. Por que el gateway es la unica entrada publica

Porque evita:

  • exponer varios puertos de negocio al usuario final

  • obligar al frontend a conocer topologia interna

  • mezclar HTTP y HTTPS de forma inconsistente

5.11.2. Por que webapp no publica un puerto propio en Compose

Porque en el despliegue integrado la web debe servirse a traves del gateway.

5.11.3. Por que hay dos Mongo

Porque el dominio de autenticacion y el de estadisticas no tienen por que compartir ni datos ni ciclo de vida.

5.12. Lo que aun no es despliegue "cloud-ready"

Es una arquitectura distribuida, pero no una plataforma de alta disponibilidad completa.

Limites actuales:

  • gamey mantiene estado vivo solo en memoria

  • no hay cola externa ni cache distribuida

  • no hay coordinacion multi-instancia de matchmaking

  • no hay estrategia de replicas para sesiones activas

Es perfectamente valido decirlo asi:

la arquitectura esta muy bien orientada a desarrollo y demostracion funcional de microservicios, pero para escalar horizontalmente el online habria que externalizar el estado vivo o introducir afinidad de sesion

6. Claves Para Presentar

6.1. Mensaje corto del proyecto

Si te piden una explicacion breve:

Yovi es una aplicacion web de Game Y organizada en microservicios. El gateway unifica el acceso, auth gestiona identidad, gamey concentra el motor del juego y el online, y stats persiste historial y agregados.

6.2. Mensaje corto del online

El online funciona con matchmaking en memoria, polling desde el cliente, tokens por jugador y workers asincronos en gamey para emparejar, vigilar inactividad y forzar cambios de turno por timeout.

6.3. Guion de 2 minutos

Puedes seguir este orden:

  1. Que es

    Es una SPA para jugar a Game Y contra bots, en local o online, con cuentas e historial.

  2. Como entra el usuario

    Todo entra por el gateway, que hace de puerta publica unica.

  3. Como se reparte

    auth_service resuelve identidad, gamey lleva el dominio del juego y stats guarda la memoria historica.

  4. Donde esta la dificultad

    La parte compleja esta en gamey, porque mantiene partidas online en memoria, genera tokens, empareja jugadores y vigila tiempos.

  5. Como se guardan los datos

    Las partidas activas viven en memoria en gamey; los usuarios viven en mongo-auth; el historial y agregados, en mongo-stats.

  6. Que tiene de interesante

    No es solo CRUD: hay coordinacion asincrona, reglas de juego, bots, matchmaking y observabilidad.

6.4. Guion de 5 minutos

Si te dejan desarrollar mas:

  1. Empieza por la topologia general.

  2. Explica el gateway como entrada unica.

  3. Separa muy claramente autenticacion, juego vivo y estadisticas.

  4. Dedica tiempo especial a gamey.

  5. Explica que el online usa polling y workers asincronos, no WebSockets.

  6. Cierra con persistencia, monitorizacion y limites de escalado.

6.5. Fortalezas reales que si puedes vender

  • La separacion por responsabilidad es clara y facil de defender.

  • gamey concentra de forma coherente motor, bots y coordinacion online.

  • El estado vivo y el historico estan claramente separados.

  • El gateway simplifica acceso, despliegue y exposicion publica.

  • El proyecto tiene observabilidad basica con Prometheus y Grafana.

  • La SPA reanuda matchmaking y partidas, lo que mejora mucho la experiencia.

  • El historial guarda tambien el tablero final, no solo contadores.

6.6. Limitaciones honestas que conviene reconocer

  • El estado vivo del online esta en memoria; si gamey cae, se pierden partidas activas y tickets.

  • El online usa polling; es mas simple que sockets, pero menos inmediato y mas costoso en peticiones.

  • Escalar varias replicas de gamey no seria trivial sin compartir estado.

  • Las partidas terminadas siguen en memoria en gamey despues de reportarse.

  • Hay un modelo de identidad algo hibrido entre sesion de frontend y autenticacion real segun el camino de acceso.

Reconocer limites no te debilita. Te hace sonar serio.

6.7. Preguntas que te pueden hacer

6.7.1. Por que microservicios y no un monolito

Respuesta util:

Porque las responsabilidades estan muy diferenciadas. Autenticacion, motor del juego y estadisticas tienen ciclos de cambio distintos. Separarlos hace mas facil razonar sobre cada dominio y desplegarlos de forma independiente.

6.7.2. Por que Rust en gamey

Porque es el servicio con mas logica y coordinacion interna. Concentrar ahi reglas, bots, estado vivo y workers asincronos encaja muy bien con una implementacion en Rust.

6.7.3. Por que no se guarda cada movimiento en base de datos

Porque el proyecto separa juego vivo de memoria historica. Guardar cada movimiento aumentaria complejidad y latencia. La implementacion actual solo persiste el resumen final y el tablero final cuando termina la partida.

6.7.4. Como evitais que un jugador actue por el otro en online

Las partidas de matchmaking generan un token por jugador. Gamey valida ese token y ademas comprueba que corresponde al turno actual antes de aceptar ciertas acciones.

6.7.5. Como se emparejan los jugadores

Hay una cola de tickets en memoria y un worker asincrono que la revisa periodicamente. Busca dos tickets compatibles por tamano de tablero y evita emparejar la misma identidad consigo misma.

6.7.6. Que pasa si un jugador se desconecta

Gamey va registrando la ultima vez que cada jugador online da senales. Otro worker revisa esa informacion cada segundo. Si alguien supera el umbral de inactividad, la partida termina por abandono.

6.7.7. Que pasa si se acaba el tiempo de turno

El monitor de gamey fuerza un pass automatico del turno. El frontend tambien muestra una cuenta atras sincronizada para que el usuario entienda lo que esta ocurriendo.

6.7.8. Donde esta la seguridad

Auth protege credenciales con bcrypt y JWT. Stats protege su endpoint interno de escritura con un token de servicio. En online, gamey protege acciones sensibles con player tokens.

6.7.9. Que papel tiene el gateway aparte de reenviar

Ademas de proxy, centraliza errores de servicios caidos, puede actuar como borde HTTPS y expone una API externa agregada con documentacion Swagger para consumidores externos.

6.7.10. Cual es el mayor limite arquitectonico

La dependencia del estado en memoria dentro de gamey para el online. Simplifica mucho la implementacion, pero hace mas dificil tolerar reinicios o escalar horizontalmente.

6.8. Frases potentes para sonar seguro

  • La autoridad del estado de juego esta en gamey, no en el cliente.

  • El gateway simplifica el acceso, pero no secuestra la logica de negocio.

  • El online esta modelado como coordinacion periodica, no como push en tiempo real.

  • Stats no participa en la jugada actual; conserva la memoria del sistema.

  • La arquitectura separa muy bien identidad, estado vivo e historial persistente.

6.9. Si quieres rematar muy bien una respuesta

Una forma muy buena de cerrar casi cualquier explicacion es:

La implementacion prioriza claridad arquitectonica y separacion de responsabilidades. Donde mas se nota es en gamey: ahi conviven el motor del juego, la coordinacion online y la integracion con estadisticas, mientras el resto de servicios quedan mas enfocados y faciles de razonar.

6.10. Resumen final para ti

Si solo tuvieras que memorizar cinco ideas, yo me quedaria con estas:

  1. Todo entra por el gateway.

  2. auth_service autentica, pero no juega.

  3. gamey es el cerebro del sistema.

  4. stats recuerda lo que ya termino.

  5. El online funciona con estado en memoria, polling y workers asincronos.

Con esas cinco ideas bien defendidas, ya puedes construir casi cualquier respuesta alrededor.
