== Runtime View


ifdef::arc42help[]
[role="arc42help"]
****
.Contents
The runtime view describes concrete behavior and interactions of the system's building blocks in form of scenarios from the following areas:

* important use cases or features: how do building blocks execute them?
* interactions at critical external interfaces: how do building blocks cooperate with users and neighboring systems?
* operation and administration: launch, start-up, stop
* error and exception scenarios

Remark: The main criterion for the choice of possible scenarios (sequences, workflows) is their *architectural relevance*. It is *not* important to describe a large number of scenarios. You should rather document a representative selection.

.Motivation
You should understand how (instances of) building blocks of your system perform their job and communicate at runtime.
You will mainly capture scenarios in your documentation to communicate your architecture to stakeholders that are less willing or able to read and understand the static models (building block view, deployment view).

.Form
There are many notations for describing scenarios, e.g.

* numbered list of steps (in natural language)
* activity diagrams or flow charts
* sequence diagrams
* BPMN or EPCs (event process chains)
* state machines
* ...


.Further Information

See https://docs.arc42.org/section-6/[Runtime View] in the arc42 documentation.

****
endif::arc42help[]

This section documents the runtime scenarios that are most useful to understand Yovi: authentication, game creation, multiplayer synchronization, statistics reporting and degraded behavior.

=== Scenario Selection

[cols="1,3",options="header"]
|===
|Scenario | Why it is relevant

|Login and session recovery
|Shows the complete path for authentication and explains how the webapp restores a valid session after reload.

|Create a new game
|Explains how the main business capability starts and where the live game state is stored.

|Matchmaking and online turn supervision
|Captures the polling flow, token-based online play and the background timeout behavior of `gamey`.

|Finish a match and update statistics
|Shows the integration between `gamey` and `stats`, including graceful degradation.

|Gateway upstream failure
|Documents the standardized error path exposed to the browser when an internal service is unavailable.
|===

=== Login and Session Recovery

[plantuml,"06_runtime_login_recovery",png]
----
@startuml
title Runtime 1 - Login and session recovery

actor Player
participant "Webapp" as Webapp
participant "Gateway" as Gateway
participant "Auth Service" as Auth
database "Mongo Auth" as MongoAuth

Player -> Webapp: Submit username and password
Webapp -> Gateway: POST /auth/login
Gateway -> Auth: POST /login
Auth -> MongoAuth: find user
MongoAuth --> Auth: user + password hash
Auth -> Auth: compare password\nsign JWT
Auth --> Gateway: 200 { token, user }
Gateway --> Webapp: 200 { token, user }
Webapp -> Webapp: Persist session locally

== Later, after reload ==
Webapp -> Gateway: GET /auth/verify
Gateway -> Auth: GET /verify
Auth -> Auth: validate JWT
Auth --> Gateway: 200 { valid, user }
Gateway --> Webapp: 200 { valid, user }
Webapp -> Webapp: Restore authenticated state
@enduml
----

* The gateway is the only public access point used by the browser.
* `auth_service` owns credential validation and token issuance.
* The browser keeps only session data; account data remains in `Mongo Auth`.

=== Create a New Game

[plantuml,"06_runtime_create_game",png]
----
@startuml
title Runtime 2 - Create a new game

actor Player
participant "Webapp" as Webapp
participant "Gateway" as Gateway
participant "Gamey Service" as Gamey

Player -> Webapp: Create game
Webapp -> Gateway: POST /api/v1/games
Gateway -> Gamey: POST /v1/games + x-user-id
Gamey -> Gamey: validate request
Gamey -> Gamey: create in-memory GameSession
Gamey -> Gamey: register active user if present
Gamey --> Gateway: 200 GameStateResponse
Gateway --> Webapp: 200 GameStateResponse
Webapp -> Webapp: Persist resumable game state
Webapp --> Player: Open game view
@enduml
----

* Active matches are stored in memory inside `gamey`.
* Creating a game does not require a database write.
* The webapp stores enough local state to resume the match later.

=== Matchmaking and Online Turn Supervision

[plantuml,"06_runtime_matchmaking",png]
----
@startuml
title Runtime 3 - Matchmaking and online turn supervision

actor "Player A" as PlayerA
actor "Player B" as PlayerB
participant "Webapp A" as WebappA
participant "Webapp B" as WebappB
participant "Gateway" as Gateway
participant "Gamey Service" as Gamey
participant "Matchmaking worker" as Worker
participant "Timeout monitor" as Monitor

PlayerA -> WebappA: Start matchmaking
WebappA -> Gateway: POST /api/v1/matchmaking/enqueue
Gateway -> Gamey: POST /v1/matchmaking/enqueue + x-user-id
Gamey --> Gateway: waiting ticket
Gateway --> WebappA: waiting ticket

PlayerB -> WebappB: Start matchmaking
WebappB -> Gateway: POST /api/v1/matchmaking/enqueue
Gateway -> Gamey: POST /v1/matchmaking/enqueue + x-user-id
Gamey --> Gateway: waiting ticket
Gateway --> WebappB: waiting ticket

Worker -> Gamey: pair compatible tickets
Worker -> Gamey: create game and player tokens

loop Poll ticket status
  WebappA -> Gateway: GET /api/v1/matchmaking/tickets/{id}
  Gateway -> Gamey: GET /v1/matchmaking/tickets/{id}
  Gamey --> Gateway: matched { gameId, playerId, playerToken }
  Gateway --> WebappA: matched
end

WebappA -> Gateway: GET /api/v1/games/{id}
Gateway -> Gamey: GET /v1/games/{id} + x-player-token
Gamey -> Gamey: update last seen and timers
Gamey --> Gateway: current online state
Gateway --> WebappA: current online state

loop During online play
  WebappA -> Gateway: GET or POST game action
  Gateway -> Gamey: forward request + x-player-token
  Gamey -> Gamey: validate token, update timers
end

opt A player is inactive or out of turn time
  Monitor -> Gamey: auto-pass current turn or resign inactive player
end
@enduml
----

* Matchmaking tickets and active online games are held in memory in `gamey`.
* The browser polls the ticket until it becomes `matched`, then opens the created game automatically.
* `x-player-token` binds actions to the correct online player.

=== Finish a Match and Update Statistics

[plantuml,"06_runtime_finish_and_stats",png]
----
@startuml
title Runtime 4 - Finish match and update statistics

actor Player
participant "Webapp" as Webapp
participant "Gateway" as Gateway
participant "Gamey Service" as Gamey
participant "Stats Service" as Stats
database "Mongo Stats" as MongoStats

Player -> Webapp: Play move, pass or resign
Webapp -> Gateway: POST /api/v1/games/{id}/...
Gateway -> Gamey: forward request
Gamey -> Gamey: apply action and evaluate match state

alt Match finished
  Gamey -> Stats: POST /internal/v1/matches/finished + x-service-token
  Stats -> MongoStats: upsert player_matches
  Stats -> MongoStats: upsert player_stats
  Stats --> Gamey: 202 processed summary
else Match still active
  Gamey -> Gamey: keep session in memory
end

Gamey --> Gateway: GameStateResponse
Gateway --> Webapp: GameStateResponse
Webapp --> Player: Show updated game status

opt Stats service fails
  Gamey -> Gamey: log warning and keep user response
end
@enduml
----

* Statistics are updated only when a match reaches a finished state.
* The reporting call is internal and protected with a service token.
* A stats failure does not block the player from receiving the final game state.

=== Gateway Upstream Failure

[plantuml,"06_runtime_gateway_failure",png]
----
@startuml
title Runtime 5 - Gateway upstream failure

actor Player
participant "Webapp" as Webapp
participant "Gateway" as Gateway
participant "Internal service" as Upstream

Player -> Webapp: Trigger action
Webapp -> Gateway: HTTP request
Gateway -> Upstream: forward request
... upstream unavailable ...
Gateway --> Webapp: 502 { message: "Bad Gateway" }
Webapp --> Player: Show request error
@enduml
----

* The gateway converts connection problems into a predictable HTTP `502` response.
* The browser does not need to know which internal service failed.
